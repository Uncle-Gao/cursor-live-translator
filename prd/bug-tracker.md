# Cursor 汉化 V2 方案缺陷与 Bug 追踪池 (Bug Tracker)

本文档用于专门记录并跟踪 V2 开发周期中遇到的疑难杂症、未解析表现以及预选的应对策略。

---

## 🐞 Bug-001：底层悬停气泡（Tooltip）未能成功触发翻译

### 📌 缺陷描述
经过 M1 阶段运行引擎 `cursor-i18n-runtime.js` 的注入与测试，设置界面的基础文本节点文本（如界面的 Label、按钮等）翻译验证成功。但在用户鼠标悬停在帮助图标、带提示信息的选择框上的气泡（Tooltip）元素时，即便底层代码中加入了针对子孙元素原生 `title` 属性的嗅探和更新拦截，系统仍然挂起，悬停气泡始终呈现未翻译的默认英文。

### 🔎 更深层的可能原因 (Root Causes)

1. **使用了动态渲染的浮动组件而非原生 `title`**：
   在早期的验证里，我们主观以为 Cursor 设置系统的所有附注都来自低复杂度的原生 `title="..."` 属性绑定。但现实中，基于 Monaco 编辑器或复杂 Web 框架下的桌面应用，为了统一在不同系统下的弹出层视觉表现，往往会**干掉原生 title**，并在鼠标 `hover` 的毫秒级瞬间，在 `<body>` 或根布局的末尾暴力生成一个绝对定位（absolute）或浮动的自定义 HTML 气泡控件（例如：`.monaco-hover` 或类似于 `.tippy-box` 这样的组件类），随后向内部填充 `textContent` 这段文本。所以此时专门去解析挂在原输入框身上的 `title` 显然会扑空。

2. **Shadow DOM（影子挂载区）的壁垒隔离**：
   如果 Cursor 的这块设置 UI 组件引入了标准 Web Component 的 `<shadow-root>` 开槽机制。由于普通的 `MutationObserver` 默认受限于挂载节点且只能监听所谓的逻辑 DOM 子树（Light DOM），它是绝对**无法穿透刺探到** Shadow DOM 内部任何发生的节点新增或属性变化的。

3. **变异的 DOM 树数据绑定属性**：
   文本并没有寄托在我们观察过滤的 `attributeFilter: ['title']` 中。前端框架非常喜欢将此类气泡数据通过自定义组件绑定，它的真实数据也许存放于 `aria-label`、`data-original-title` 等属性，随后再触发气泡层直接调用对应的字符串。

4. **文本无法严丝合缝匹配（含空格或转义符）**：
   我们字典映射采用了粗暴的精确等值判断（虽然经过了 `trim()`），但若是气泡组件生成的文本中穿插了其他 HTML Tag（如 `<span></span>`）或是含有额外的缩进、换行与连字符 `&nbsp;`，严格模式比对必然直接滑过判定从而导致没有翻译。

### 💡 候选解决方案 (Possible Solutions)

为了彻底拿下这个顽疾，我们需要尝试在后续采取以下解法应对：

- **方案甲：拦截 Custom Floating Hover Node（最佳兼容尝试）**
  不必拘泥于寻找静态标签上的属性。既然任何内容出现在页面上一定是产生了新标签，只要打开 Chrome DevTools，按下鼠标悬停使它浮现并**冻结调试端点**(快捷键 `Debugger`)，精确揪出那个浮现出来的包裹节点的 `class` 特征名（比如名叫 `.monaco-hover`）。然后将其纳入 M1 原本针对 `textContent` 子树更新的扫描逻辑下。因为它本质就是个装有文本节点的新生 DOM，触发 `childList` 时也能将其拿住翻译。

- **方案乙：拓宽属性拦截黑名单网络**
  如果确认没有悬浮元素动态生成，而是直接挂在元素上，我们可以扩大侦听域，在 MutationObserver 中追加监听 `aria-label` 以及各类以 `data-` 开始的框架级气泡绑定属性群组，进行同等规格的映射替换。

- **方案丙：模糊查找匹配机制**
  对拦截到的较长字符串进行模糊去噪，使用首尾空白符清理正则替换掉多余的标点，或者把绝对的键值对匹配退化为长短句特征识别，防止任何细小的隐形富文本阻拦我们对气泡英文字符的识别。
