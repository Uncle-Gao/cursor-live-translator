# Cursor 汉化 Pro：AI 驱动的实时本地化引擎 (V2.1 PRO)

本项目是专门为 Cursor 定制的高性能本地化引擎。通过 **V2.1 Trusted Bootstrap Injection（受信任引导挂载）** 架构，实现了离线字典匹配、动态变量正则解析与多引擎 AI 实时翻译的深度融合，为您提供全方位的中文开发体验。

> [!IMPORTANT]
> **全系架构跨代升级：运行时 DOM 拦截与受信任引导挂载**
> 这是 V2 PRO 架构相较于传统 V1 版本（以及市面常见汉化插件）的革命性突破。为解决暴力修改混淆源码“逢更新必黑屏”的致命缺陷，V2.1 PRO 完全舍弃了静态正则替换文件的方式。我们将汉化引擎作为外挂系统，直接**内联追加（Append）至受信任的 `workbench.js` 引导程序尾部**。随后利用 `MutationObserver` 在内存渲染层实时汉化 UI。这确保了补丁在最新版 Cursor 的高安全性环境下依然能 100% 静默启动，无报错、不伤源码、稳定防崩溃。

- **精准映射**：将捕获组实时填入汉化模板（如：`找到 $1 个文件`）。
- **极速响应**：匹配结果进入三级缓存系统，确保在界面高频渲染下仍能维持毫秒级响应。

## 🏗️ 进化史：从 V1 到 V2.1 PRO 架构演进

如果你曾使用过早期的 V1 版本，你可能经历过“更新即失效”或“启动即黑屏”的情况。而即便在初版 V2 时代，也同样面临过安全审核收紧的挑战。现役的 **V2.1 PRO 架构** 正是为了终结这些痛点而生。

| 特性 | V1 (静态文本替换) | V2.0 (早期 HTML 注入) | V2.1 PRO (受信任引导挂载) |
| :--- | :--- | :--- | :--- |
| **稳定性 (Stability)** | **极高风险**。强行正则修改数 MB 的混淆核心代码，极易造成语法破坏导致黑屏。 | **存在隐患**。对 `workbench.html` 直接插入外部 script 标签。 | **稳如磐石**。利用原生受信任上下文，仅在 `workbench.js` 尾部追加逻辑。 |
| **汉化深度 (Coverage)** | **仅限静态文本**。只能翻译安装时存在于文件中的死词条。 | **动态覆盖**。通过 MutationObserver 实时监听 DOM，但处理不够平缓。 | **高保真动态**。三级引擎（字典+正则+AI）全天候翻译 AI 与组件，配套 RAF 防抖零延迟。 |
| **安全审计 (Security)** | **频繁警告**。极易破坏文件原有系统签名。 | **被安全策略阻截**。在新版 Cursor 中，外部 HTML 注入极易触发 `Trusted Types` 异常和 CSP 拦截。 | **免报免拦截**。直接从受信任信任链（JS 引擎自身）挂载，完全不触发控制台警告安全红线。 |

### 🚀 为什么要彻底放弃 V1？
1. **文件规模危机**：随着 Cursor 功能增强，其 `main.js` 文件越来越庞大，V1 的“查找替换”模式速度慢且错误率呈几何级上升。
2. **动态内容爆发**：V1 无法汉化运行时产生的 AI 对话、新版 Composer 等动态 UI，这在 AI 编辑器中是致命的。
3. **安全审计升级**：Cursor 0.45.x+ 引入了严格的哈希校验与 CSP 限制，传统的 V1 修改方式会触发各种系统报警，甚至导致签名锁定。

## 🛠️ V2 PRO 技术原理深度解析

V2 PRO 并非简单的字符串替换工具，而是一套运行在 Cursor (Electron) 渲染进程内部的**实时 UI 翻译系统**。

### 1. 受信任引导挂载 (Trusted Bootstrap Injection)
不同于传统的 HTML 注入，我们将内联脚本追加至 `workbench.js` 尾部。由于其属于原生信任链的一部分，汉化逻辑被视为“绝对受信任”，完美绕过 `Trusted Types` 安全审计与内容安全策略 (CSP)。

### 2. DOM 语义级监听 (MutationObserver)
利用浏览器原生的 `MutationObserver` 接口，全量监听 `document.body` 的变动。无论是 AI 对话、新弹出的 Composer 浮窗，还是鼠标悬停提示，都能在渲染瞬间被捕捉并汉化。

### 3. 三级级联匹配引擎 (Triple-Tier Lookup)
- **L1 离线字典**：秒级检索已知 UI 静态词条。
- **L2 正则映射 (M4)**：识别带数字、百分比的动态模式（如 `Found 5 files`），一套规则适配无限场景。
- **L3 AI 增强**：针对字典和正则无法覆盖的复杂长句，自动调用多引擎接口实现高保真翻译。

### 4. 极致性能设计 (Performance Optimization)
- **RAF 防抖**：渲染逻辑挂载在 `requestAnimationFrame` 下，确保仅在空闲帧执行，零 CPU 震荡。
- **二级缓存体系**：将结果持久化至内存与 `localStorage`。同一个动态词条只需翻译一次，重启后秒开。
- **异步批处理**：AI 翻译请求采用 2000ms 批处理策略，自动合并并发需求，节省 Token 并消除延迟感。

## 📂 项目结构
- **`cursor_zh_pro.js`**: 核心安装器 & 持久化配置中心。
- **`cursor-i18n-runtime.js`**: V2.1 运行时引擎，负责 DOM 监听、正则模式匹配与 AI 调度。
- **`i18n/`**: 
  - `dictionary.json`: 结构化词条仓库（含离线直接匹配与 Regex 模式匹配）。
- **`~/.cursor_zh_pro/`**: 用户本地加密存储的 API 配置资产目录。

## 🎯 汉化作用域与架构边界 (Translation Coverage)

因为采用的是基于 `document.body` 的 `MutationObserver` 全局 DOM 拦截架构，本项目不局限于单个静态页面，而是**接管整个前端 UI 界面**。

### 🟢 深度覆盖区域
- **高阶 AI 面板**：[Cursor Chat] 聊天区、[Composer] 多文件智能体弹窗、[Cmd + K] 行内命令框。AI 的长篇英文流式输出都会被引擎“按句捕获”并实时汉化。
- **全局配置与设定**：Cursor 的 Features 开关面板、Models 模型配置界面。
- **主界面与弹窗**：顶部原生菜单、搜索与文件资源管理器、浮动 Notify 警告通知栏、悬停代码提示（`title` 级汉化）。

### 🔴 安全隔离区（刻意绕过的领域）
为了绝对保障您的代码安全，汉化运行时 (`cursor-i18n-runtime.js`) 中内置了极其严格的 `SKIP_TAGS` 安全白名单：
- **您编辑的代码区**：严禁侵犯。引擎在嗅探 DOM 树时，会自动过滤 `<pre>`、`<code>` 及部分特定层级的文本流，**不会有任何一个字符的代码被误翻译**。
- **可输入控件**：您正在敲字的 `<textarea>` 和 `<input>` 输入框。
- **Terminal 终端**：基于底层进程直连渲染的纯终端回显区域。

简而言之：**只要是 Cursor 想用英文弹窗给您看的，全部汉化；只要是属于您的私有代码和命令执行区，寸步不染。**

## 🚀 快速开始

### 1. 环境准备
确保已安装 Node.js 环境。

### 2. 运行安装程序
在任意终端执行：
```bash
node cursor_zh_pro.js
```
*(为方便以后随时唤起控制台，您也可以自行在 `~/.zshrc` 中为该命令设置 `cursor-zh` 的 alias)*

> 💡 **全面兼容自定义安装路径**：无论您使用的是便携版、还是将应用安装在非系统盘（如 `D:\MyApps\`），本脚本都完全支持！若脚本未在默认系统路径嗅探到 Cursor，它会自动悬停并请您输入（或直接将文件夹拖拉进终端）您的自定义安装路径，随后智能寻址并完成深层挂载。

### 3. 操作指引
1. 选择 `3. 配置 AI 翻译引擎`（如 DeepSeek），录入您的 API Key 并通过在线连通性测试。
2. 选择 `1. 一键汉化`。
3. 彻底重启 Cursor，享受顶级 AI 汉化。

## 🛡️ 安全性与隐私 (Security & Privacy)

针对用户对 AI API 安全性及隐私的考量，在此做出明确说明：

### 1. API Key 本地化存储
- **物理隔离**：您的 API Key 既不嵌入代码，也不随本项目 Git 提交。它们被持久化存储在您本地系统的家目录 `~/.cursor_zh_pro/config.json` 之中。
- **防止误传**：存储位置位于项目路径之外，从源头上杜绝了因误执行 `git push` 导致泄露的风险。

### 2. 直连通信，无中间商
- **点对点传输**：所有的翻译请求均由您的 Cursor 渲染进程（Renderer Process）**直接发往** AI 官方 API 接口（如 `api.deepseek.com`）。
- **无中转服务器**：本项目不设立任何后端代理或数据中转站，不存在“二房东”窃取或拦截 API 流量的可能。

### 3. 数据隐私边界
- **仅限 UI 级嗅探**：汉化引擎仅对 `document.body` 中的文本节点进行读取与匹配。
- **不触碰源码**：本项目不会主动读取您的工程代码文件。翻译的范围严格限定在编辑器自身的 UI 界面、菜单栏、对话框及 AI 聊天的展示文本。
- **代码可审计**：核心运行时 `cursor-i18n-runtime.js` 仅约 200 余行代码，逻辑透明。任何人均可随时审计其网络请求行为。

### 4. 最小化 Token 消耗
- **双重缓存**：通过内存与本地存储二级缓存，同样的 UI 词条在您的机器上终生只需翻译一次，最大程度降低了 API 的曝光频次与资费开销。

## 🛠️ 常见问题 (Troubleshooting)

### 1. 汉化后 Cursor 打不开了（提示损坏）？
这通常是 macOS 的签名权限校验。安装程序通常会自动修复，如果因权限无法重签，请手动执行：
```bash
sudo xattr -cr /Applications/Cursor.app
sudo codesign --force --deep --sign - /Applications/Cursor.app
```

### 2. “恢复官方原版”选项是做什么的？
“恢复官方原版”旨在随时移除汉化补丁。系统会自动检测并使用之前为您预先隔离好的对应原生版本进行物理覆盖，同时清除多余的缓存和垃圾文件。


### 3. 如何知道汉化插件是否成功运行了？
有两种方式可以明确验证引擎是否存活：
1. **看安装器状态**：启动 `cursor_zh_pro.js` 面板，如果您的“备份状态”显示为绿色 `🟢 [版本匹配] v{当前版本号} 引导备份就绪`，说明已经成功挂载。
2. **看底层控制台**：在 Cursor 中按下 `Cmd + Option + I` (Mac) 或 `Ctrl + Shift + I` (Win) 打开开发者工具并切换到 `Console` 标签页，如果您在开头看到了紫色的印记 `[Cursor-i18n] 多引擎动力系统就绪 | 当前模式: xxx`，则代表 V2.1 PRO 已完美生效！

### 4. 怎么确认 AI 翻译（信达雅增强）是真的在处理，而不是只用了离线字典？
由于高频的基础 UI 全由离线字典（L1）和正则引擎（L2）在 `0ms` 内瞬间处理掉，为了专门验证作为后备大底的 AI 引擎（L3）是否在辛勤劳作，您可以：
1. **Network 抓包**：同样在开发者工具中，切换到 `Network` (网络) 标签页。然后故意让 Cursor 弹出一些未见过的报错，或者随便问 Composer 一个冷门报错。此时您会在 Network 列表里看到一条发往 `api.deepseek.com`（或您配置的 endpoint）的异步并发请求，这就是 AI 正在执行实时翻译！
2. **翻译延迟体感**：在 Cursor 渲染出全新的大段英文句子时，由于引擎采取了 `2000ms` 的批处理防抖机制以节省您的 Token，您会肉眼观察到这部分英文在屏幕上停留了约 2 秒钟，然后“唰”地一下集中变成优雅的中文——这正是 AI 翻译引擎独有且迷人的呼吸感。
